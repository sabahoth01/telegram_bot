## Что такое BigData

**BigData** — данные, не помещающиеся в оперативную память одной рабочей машины.

### Примеры BigData
- Пользовательские логи
- Датасеты для ML
- Мультимедиа
- Сырые данные (raw): фотографии, временные ряды с экспериментов, показания датчиков
- Графы
- Дампы, версионирование
- Географическая информация
- Синтетические данные
- **Самые большие** — поисковый индекс (веб-страницы + метаинформация)

> 💡 Kiwix — приложение для скачивания и офлайн-чтения веб-страниц (например, Википедии)

## Способы хранения BigData

Разделены по способам взаимодействия и парсинга:

### Файловые системы и протоколы
- **Жёсткий диск / SSD** — сырой формат
- **sshFS** — библиотека, монтирующая на машину-исполнитель файловую систему удалённого хоста и организующая запрос-ответ по SSH + кэш (даёт прирост ~x10 к скорости)
- **RAID-массив** — файловые системы с избыточностью

### Базы данных

**Реляционные:**
- PostgreSQL
- SQLite

**Нереляционные:**
- Графовые
- Документные
- Векторные
- Комбинированные
- Key-value: **Redis**
- **Колоночные БД** (предоставляют интерфейс РБД): Vertica, **ClickHouse**, Greenplum

**Распределённые:**
- Блокчейн, торрент, одноранговые сети
- **HDFS** — Hadoop Ecosystem
- **YandexTablesSaurus (YT)**
- **YQL** — язык запросов для YT

**Файловые системы для больших файлов:**
- DFS, VCS, Git LFS
- **S3 bucket storage**

> 💡 Внутри БД данные можно сильно сжимать. Колоночные БД сжимают по умолчанию, строковые — если дополнительно попросить.

## Форматы хранения данных на диске

### Несжатые (raw)
| Формат | Примечания |
|--------|-----------|
| `txt` | Простой текст |
| `csv` / `tsv` / `dsv` | Парсеры несовместимы из-за запятых с кавычками; `tsv` решает проблему через табуляцию: `value1 \t value2` |
| `json` | Массив dict vs «dict \n» (jsonl); второй (jsonl) значительно быстрее |
| `jsonl` | Построчный JSON — каждая строка отдельный объект |
| `xml` / `yaml` | Некоторые парсеры работают лучше, чем json |
| Бинарники | `.bin`, `.exe`, `.npy` (хранение векторов/массивов numpy) |

### Сжатые
| Формат | Сжатие | Примечания |
|--------|--------|-----------|
| `parquet` | ~20x | Колоночный |
| `orc` | ~20x | Лучше parquet (см. ниже) |
| `xlsx` | — | zip-архив фиксированной структуры (новый формат MS Office) |
| `pickle` | ~20x | Только Python |
| `zip` / `gz` / `tar` | ~20x | Универсальные архивы |
| `hdt` | — | Для RDF-данных |
| `snappy` | — | Быстрое сжатие, используется внутри Parquet/ORC |

> В zip, orc, parquet, pickle ожидается сжатие в **~20 раз**

## Формат ORC (Optimized Row Columnar)

Сочетает плюсы **построчного** и **поколоночного** хранения таблиц.

### Структура хранения
- **Stripe** — набор строк с footer внутри. Каждый stripe преобразуется в поколоночное хранение
- **Header / Footer** — метаинформация о стрйпах
- **Метрика** для быстрого поиска

### Поддерживаемые операции
- Поиск по ключу
- Добавление / удаление данных (в начало, конец, произвольное место)
- Фильтры (предикаты, индекс-фильтр)
- Потоковое чтение
- SQL / SQL-like интерфейс: группировки, сортировки

### Группировка в ORC
- Сортировка по ключу группировки + проход подряд
- Хэш-таблицы

### Оптимизация сжатия
Stripe выбираются особым образом для максимального сжатия (например: одно значение / его номер вместо дубликатов в каждой строке).

### Сравнение
> ORC **быстрее и лучше**, чем Parquet

### Работа в Python
```python
import pyorc
```

## Python-инструменты для работы с данными

### Pandas (устарел морально)
- Написан на C/C++
- Медленнее современных альтернатив

### Polars — «убийца Pandas»
- Написан на **Rust** (`.rs` — стандарт расширения файлов Rust)
- Читает `xlsx` в **~10 раз быстрее** Pandas
- Использует специальные библиотеки для хранения/чтения, в т.ч. `python-calamine`
- Сложные функции (VBA в Excel, App Script в Google Sheets) может не прочитать

#### Работа в Polars — SQL-like подходы:
```python
# 1. Raw SQL — когда не нужно разбираться с конкретными разными API
#    нужна масштабируемость

# 2. Через объекты (SQLAlchemy) — для большого числа однотипных запросов
```

**Важные правила:**
- Смешивать подходы **нельзя**
- В SQL-запросах **нельзя делать вложенные запросы** — только через переменные, хранящие образы таблиц. Причина: низкая читаемость и достаточно высокая степень оптимизации

### DuckDB
Библиотека для **raw SQL в pandas.DataFrame**.

**Принцип работы:**
1. DataFrame парсится как строка
2. Из строки `locals()` / `globals()` обращаются к ней как к переменной через `globals.get()`

```python
import duckdb
result = duckdb.sql("SELECT * FROM my_df WHERE age > 18")
```

> 💡 Полезно прочитать документацию DuckDB для прокачки навыков SQL

## Многопоточность в Python

### Главная проблема — GIL
**GIL (Global Interpreter Lock)** — блокировка, разработанная Гвидо ван Россумом, запрещающая переход в накладной многопоточный интерпретатор + сборщик мусора, пока он не побьёт однопоточный.

- В **Python 3.14** GIL можно отключить при локальной сборке языка
- **PEP (Python Enhancement Proposal)** — включает кодстайл и предложения по улучшению языка

### Когда применяется многопоточность
- I/O-операции (чтение/запись файлов, сетевые запросы)

### Процессы vs Потоки

| Характеристика | Процессы | Потоки |
|---------------|----------|--------|
| Уровень | Логическая верхнеуровневая сущность | Низкоуровневая ядерная |
| Изоляция | Изолированы друг от друга (память, ресурсы) | Могут иметь общую память |
| Коммуникация | Через общую память, создаваемую вручную с контролем | Общая память |
| Идентификация (Linux) | По `pid` | — |

**Важно:** Python запускается единым потоком и поэтому разбивается на **процессы**, в каждом из которых — отдельный интерпретатор.

### Библиотеки

```python
import multiprocessing as mp                       # настоящие процессы
from concurrent.futures import ThreadPoolExecutor  # НЕ настоящие потоки — исполняются последовательно из-за GIL
```

**Настоящие потоки используются для:**
- Input/Output (считывание + вывод данных)
- CPU (обработка данных)

### Корутины
- Лёгковесные (по вычислительным мощностям) потоки
- В Python их нет, но потоки Python очень похожи на них
- Сильно эксплуатируют особенности железа
- Вычислительно почти бесполезны (как и потоки)
- Лучше использовать только для input/output

### Sub-интерпретаторы (Python 3.14)
Внутри 1 процесса и 1 потока — запуск нескольких Python-интерпретаторов, каждый из которых делает что-то своё.

Подключение:
1. При сборке Python
2. Импортом модуля в Python

## JIT-компиляция

**JIT (Just-In-Time)** — сброс интерпретатором циклов и сложных конструкций в C++-бинарь вместо цикличного прохода по коду цикла.

**Преимущество:** сильная экономия времени при обработке данных, особенно циклов.

### Numba

```python
import numba

@numba.jit()   # не упадёт при некомпилируемых конструкциях
def func(): ...

@numba.njit()  # nopython=True — упадёт при некомпилируемых, но это максимальная оптимизация
def func(): ...
```

**Отличие `jit` от `njit`:** `nopython=True` позволяет обрабатывать непарсируемые абстракции, но `jit` не упадёт при некомпилируемых конструкциях, а `njit` упадёт. `njit` лучше, так как это максимальная оптимизация.

### Что компилировать
✅ Большие циклы + рекурсия

### Что НЕ нужно компилировать
❌ Малые циклы (~3 итерации)  
❌ Примитивы C++  
❌ bind на C++

### Требования к коду для JIT
- Типизация
- Примитивы (всё сложнее списка обрабатывает с трудом, **словари не обрабатывает**)

### Что НЕ компилируется
- Обращение к переменной через её строку (имя)
- Словари (любой класс)

### Хранение скомпилированного кода
Скомпилированные файлы хранятся в `__pycache__`.

### Использование на Python
```python
# Cython — для C-биндингов в Python
```

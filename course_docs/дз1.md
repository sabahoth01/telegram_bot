## Задания

### 1. Найти датасет
- Размер: **> 10 ГБ** (для кластера — 50 ГБ)
- Желательно: текстовые данные, хотя бы в raw-формате
- Источники: Kaggle, HuggingFace, Common Crawl и др.

### 2. Хранение в разных форматах и бенчмарк

Замерить **cpu-time** и **wall-time** для каждого формата:

| Формат | Примечания |
|--------|-----------|
| `json` | Внимание: 1 ГБ может обрабатываться около часа |
| `jsonl` | Построчный JSON, быстрее для стриминга |
| `tsv` / `csv` | TSV предпочтительнее (нет проблем с запятыми/кавычками) |
| `parquet` | Колоночный, сжатый |
| `orc` | Optimized Row Columnar, быстрее parquet |
| `SQLite` | Реляционная БД |

### 3. SQL-подобные запросы

Реализовать следующие операции с использованием каждой библиотеки:

- **Фильтрация по предикату** (`WHERE`)
- **Группировки** (`GROUP BY`)
- **Оконные функции** (`OVER PARTITION BY`)

### 4. Используемые библиотеки

| Библиотека | Назначение |
|------------|-----------|
| `pyorc` | Работа с форматом ORC |
| `duckdb` | SQL-запросы к DataFrame |
| `polars` | Современная замена Pandas (на Rust) |
| `sqlalchemy` | ORM-стиль SQL |
| `raw python` | Чистый Python, базовый вариант |
| `raw python + оптимизации` | JIT (numba) + распараллеливание |
| `json` / `orjson` / `ujson` | Сравнение JSON-парсеров |

## Подсказки из лекций

- `tsv` лучше `csv`: нет проблем с парсингом кавычек и запятых
- ORC быстрее и компактнее Parquet
- `jsonl` (по одному dict на строку) обрабатывается значительно быстрее, чем один большой JSON-массив
- `polars` читает `xlsx` в ~10 раз быстрее `pandas`
- Используйте `numba.njit()` для максимальной JIT-оптимизации (упадёт на некомпилируемом коде — это и есть цель: максимальная оптимизация)
- В zip, orc, parquet, pickle ожидается сжатие в ~20 раз

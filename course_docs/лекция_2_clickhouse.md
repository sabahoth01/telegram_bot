## Postgres vs ClickHouse

| Postgres | ClickHouse |
|----------|-----------|
| Малонагруженные системы | Высокая нагрузка по запросам и объёму данных |
| Учебные задачи | Эффективные хранение и обработка очень больших логов |
| Простой порог входа | Более высокий порог входа |

## Типичный BigData пайплайн со Spark

Проблема: в больших командах Spark обычно на кластере с общими ресурсами. Поэтому лучше вычитать данные в ClickHouse — и обрабатывать, визуализировать там.

```
1. HDFS — хранилище
2. Spark — обработка
3. Hive — SQL-операции
```

Затем визуализация через Grafana / Superset.

## Движки (Engines) ClickHouse

### 1. MergeTree — основной движок

Для хранения данных. Обладает ключами:
- **Первичный ключ** (`primary key`) — однозначно определяет запись (часто — колонка `id`)
- **Ключ сортировки** — наиболее эффективен для сортировки и агрегации

#### Виды MergeTree
| Вид | Назначение |
|-----|-----------|
| Replacing | Заменяет дубликаты |
| Summing | Суммирует значения |
| Aggregating | Агрегирует данные |
| Collapsing | Схлопывает строки |
| Versioned Collapsing | С версионированием |
| Graphite | Для метрик Graphite |
| Coalesing | Объединение |

Виды отличаются способами обработки при замене значения / вставке нового.

#### Пример структуры таблицы логов

Столбцы:
- `timestamp` — стандартный ключ сортировки, позволяет проводить **ротацию логов** естественно
- `service` — место события (сервис, откуда пришёл лог)
- `user_id`
- `event` — конкретное действие/событие
- `object` (поисковая строка, комментарий и пр.)

> **Ротация логов** — механизм для предотвращения переполнения памяти: при исчерпании объёма логов при записи их в стек можно переписывать старые записи новыми с головы (зацикливать запись).

#### Типичные запросы к таблице логов
- DAU / MAU
- N взаимодействий с объектом (просмотров / кликов)
- Графики активности
- Время в сервисе
- Антифрод
- Рекомендательные системы и CJM (Customer Journey Map)
- Метрики качества поиска:
  - **CTR (Click-Through Rate)** — метрика эффективности клика (показ → клик)
  - **PFound**
  - Группировка элементарных событий

#### Алгоритм поиска при сортировке по ключу
1. Сортировка по первой колонке
2. Разбиение остальных итеративно по одинаковым значениям в колонках
3. Сортировка значений внутри каждой колонки
4. Бинарный поиск

### 4. Специальные движки (синтаксический сахар)

| Движок | Назначение |
|--------|-----------|
| **Distributed** | Обёртка для кластера (данные не влезают на 1 машину) |
| **Dictionary** | Key-value БД, к которой можно обращаться во время запросов |
| **File** | Интеграция с файловой системой |
| **Join** | Стыковка таблиц с разных кластеров |
| **View** | Материализованная (физически на диске) вычислимая таблица |

#### Dictionary — пример использования
```sql
-- Найти в словаре id по значению
WHERE region.id = dictGet('<name>')
-- Часто это реализуется через join, здесь — через hash-map
```

#### View (материализованные представления)
- Обычный `VIEW` — SQL-запрос, хранится в памяти как запрос
- **Материализованный VIEW** — физически лежит на диске, зависит от дочерних таблиц

ClickHouse обновляет View при обновлении дочерних таблиц:
- Автоматически при простое без задач
- Самостоятельно при запросе к View (исполняет запрос при обращении)

> ClickHouse запрещено отдавать неверные («сломанные») данные — поэтому View всегда актуален.

## Distributed Engine

Обёртка на движок для **кластера**. Полезен, когда данные не влезают на 1 машину.

- MergeTree на каждой машине должны быть одинаковые
- Структура кластера: `main → worker`
- При отсутствии явного задания main — структура кластера может меняться, обязанности main могут переходить
- При запросе **не к мастер-ноде**, а к дочерней — дочерняя выполнит его, и данные будут ограничены одной машиной

---

## Replicated Engine

Создание **реплик данных** → получение RAID-массива (полностью синхронизированных клонов с обновлением из main в периоды наименьшей загрузки сети).

Полезно для:
- Защиты данных от потери
- Защиты фрагментов данных от поломки
- Обеспечения доступа к данным

## Поддерживаемые операции

Все базовые, но **не** оконные функции (`window`).

Замена оконных функций через `lag` / `lead` — из-за поколоночного хранения:

```sql
-- Вместо оконных функций:
-- 1. for по значениям сортированной column1
-- 2. lambda к результатам column2
SELECT lagInFrame(timestamp) OVER (PARTITION BY user_id ORDER BY timestamp)
FROM user_logs;
```

## Материализованные колонки в ClickHouse

При хранении в колонке JSON как строки — создаётся **материализованная колонка** типа json:

```sql
column materialized_column
json_str json.parse(column)
```

При запросе к исходной колонке выдаётся значение материализованной:
- Синхронизированы → значение сразу
- Не синхронизированы → значение обновляется (вызывается функция) и выдаётся результат
